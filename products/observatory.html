The problem

Knowledge rots the day after you build it.

Sources change, regulations update, contracts renew — and static pipelines never notice. The Observatory watches sources continuously and turns changes into governed updates, not silent drift.

What it produces

A loop, not a batch job.

Watch targets

Scope born from what your knowledge base must keep current — entities, claims, sources, freshness requirements — plus declared discovery scopes.

Drift events

Potential knowledge changes: subject, new evidence, current claim, confidence — never a direct write.

Re-ingestion triggers

New evidence re-enters through Parser; Verifiable Knowledge compares, versions and adjudicates — then serves the new state.

How it works

Three moves. Nothing hidden.

01

Watch

Your knowledge base defines what must stay current; declared scopes add discovery. Sources are admitted by scored decision.

02

Detect & compare

The world is compared against the known state — drift becomes an evidence event, not an edit.

03

Re-enter the flow

Parser structures the new evidence; Verifiable Knowledge adjudicates and versions it, then serves the new state.

Questions

FAQ

Is the Observatory available?

Rolling out. The observation→task funnel is validated by experiment; the API surface ships behind a pilot flag. Ask for access via a pilot.

Can I buy Observatory alone?

No — it watches and updates a Verifiable Knowledge base, so it requires one. The dependency is architectural: detection without a governed place to adjudicate change would just be alerts. Verifiable Knowledge never requires Observatory.

Which sources can it watch?

Public and private: web sources, portals, feeds, drives, databases. Each is admitted individually — no source enters without a scored decision.

Does the Observatory edit my knowledge base?

Never directly. It emits drift events; new evidence re-enters through Parser and is adjudicated in Verifiable Knowledge — with versions, contradictions and history preserved — before the new state is served.
